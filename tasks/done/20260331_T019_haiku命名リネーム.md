# T019: haiku 命名リネーム（モデル非依存の命名へ）

## 背景・目的
- `news_raw.haiku_accepted` 等、コードベース全体に「haiku」という命名が約70箇所残っている
- 実際のモデルは Gemini 2.5 Flash に移行済みで、Claude Haiku は使われていない
- 誤った命名がハルシネーション（誤解・誤情報）の原因になる
- **今後モデルが変わっても問題のない、モデル非依存の名前にする**

## 命名方針
| 旧名 | 新名 | 理由 |
|---|---|---|
| `haiku_accepted` (DBカラム) | `filter_accepted` | モデル名に依存しない |
| `filterAndTranslateWithHaiku()` | `filterAndTranslateNews()` | 処理内容を表す |
| `filterAllNewsWithHaiku()` | `filterAllNews()` | 同上 |
| `translateTitlesWithHaiku()` | `translateNewsTitles()` | 同上 |
| `updateHaikuResults()` | `updateFilterResults()` | 同上 |
| `HaikuTranslatedItem` (型) | `FilteredNewsItem` | 同上 |
| `news_haiku_*` (キャッシュキー) | `news_filter_*` | 同上 |
| コメント・ログの「Haiku」 | 「Gemini Flash」or モデル名を書かない | 文脈に応じて |

## 影響範囲（約70箇所）
- DBカラム: 14箇所（本番D1マイグレーション必要）
- 関数名: 7箇所（import含む）
- 型名: 3箇所
- キャッシュキー: 3箇所
- コメント・ログ: 約30箇所
- ドキュメント: 約10箇所
- 完了済みタスク(done/): 履歴として残す（変更不要）

## 注意事項
- **DBカラム `haiku_accepted`**: 本番D1にデータがあるため `ALTER TABLE news_raw RENAME COLUMN haiku_accepted TO filter_accepted` が必要
- **`market_cache` の `news_haiku_*` キー**: 既存キャッシュは2時間で自動パージされるため、デプロイ後に自然解消
- featureブランチで作業し、PRレビュー後にマージ
- `tasks/done/` 内の過去タスクファイルは歴史的記録のため変更しない
