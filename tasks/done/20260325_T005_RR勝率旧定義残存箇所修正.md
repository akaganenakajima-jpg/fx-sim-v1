# T005: RR勝率 旧定義残存箇所の修正

## 背景
「勝率」概念統一（RR≥1.0=勝ち）の移行において、5周E2E調査により
バックエンド全体で旧定義（`pnl > 0`）が残存している箇所を網羅的に発見した。

## E2E調査方法（5周）
| 周 | 検索パターン | 目的 |
|---|---|---|
| 1周目 | `pnl > 0`, `pnl <= 0`, `pnl < 0` | 直接的な旧定義パターン |
| 2周目 | `CASE WHEN.*pnl` | SQL内のCASE WHEN条件 |
| 3周目 | `winRate`, `wins`, `isWin`, `outcome` | 勝敗計算ロジック変数 |
| 4周目 | `勝ち`, `負け`, `勝率`, `的中` | 日本語ラベル整合性 |
| 5周目 | `realized_rr` 使用箇所の逆引き | 使うべきなのに使っていないファイル特定 |

#### T003: index.ts — total_win_pnl/total_loss_pnl SQL不整合 (2箇所)
| # | 行 | 現在 | 修正後 |

### P1: 統計関数の修正

#### T007: stats.ts — rollingReturns (1箇所)
| # | 行 | 現在 | 修正後 |
|---|---|---|---|
| 1 | 92 | `slice.filter(v => v > 0).length` | 関数シグネチャ変更: pnls の代わりに `{ pnl, realized_rr }[]` を受け取るか、呼び出し元で outcomes を渡す |

**注意**: この関数は `pnls: number[]` を受け取る汎用関数。RR基準に変更するには:
- 案A: シグネチャ変更 `{ pnl: number; realizedRR: number | null }[]`
- 案B: 別パラメータ `outcomes: boolean[]` を追加
- 案C: 呼び出し元(api.ts:579)で realized_rr 基準の outcomes を別途計算して渡す

### P3: 検討事項（変更不要 or 要検討）

#### T006: api.ts — profitTop/lossTop (3箇所)
pnl > 0 / pnl < 0 で利益トップ/損失トップを分類。これは「金額として利益/損失」なので旧定義問題ではない。変更不要。
