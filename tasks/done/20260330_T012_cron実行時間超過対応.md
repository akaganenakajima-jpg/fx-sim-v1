# T012 cron 実行時間超過（110〜140秒）対応

**作成日**: 2026-03-30
**優先度**: 🔴 高
**発見経緯**: E2E Cycle 8〜10（バックエンドE2Eチェック）

---

## 背景・目的

cron の `grandTotalMs` が毎回 110〜140秒に達し、「実行時間超過」WARN が連続発生している。
Cloudflare Workers 有料プランのウォールクロック上限（15分）内ではあるが、
`newsMs`（12〜13秒）と `paramReviewMs`（最大14秒）が支配的なボトルネック。
このオーバーヘッドが PATH_B2 の AbortError（タイムアウト）の根本原因にもなっている。

## 現象

```
実行時間超過: 116733ms  fetchMs:3294 newsMs:13327 pathBMs:0 paramReviewMs:0
実行時間超過: 140040ms  fetchMs:20950 newsMs:12887 pathBMs:0 paramReviewMs:0
実行時間超過: 117880ms  fetchMs:4328 newsMs:13258 pathBMs:0 paramReviewMs:0
```

grandTotalMs の内訳（代表値）:
| フェーズ | 時間 |
|---|---|
| fetchMs | 3〜21秒 |
| newsMs | 12〜13秒（定常的に重い） |
| paramReviewMs | 0〜14秒（スパイク） |
| tpSlMs | 1〜4秒 |
| pathBMs | 0〜数秒 |

## 調査手順

1. `newsMs` の内訳を調査
   - `src/news.ts` の fetchNews / filterNews / translateNews のそれぞれの時間を計測
   - 外部APIへのリクエスト（Reuters RSS, CNBC等）のタイムアウト設定を確認
2. `paramReviewMs` がスパイクする条件を調査
   - `src/index.ts` の paramReview ロジックがどの条件で重くなるか確認
3. `fetchMs` が 20秒に達するケースの原因調査
   - どの外部APIが遅いか特定（yahooFinance / frankfurter / FRED etc.）

## 修正方針（調査結果により決定）

- newsMs 改善: ニュースフェッチを並列化、またはタイムアウトを短縮してスキップ
- paramReviewMs 改善: 重いロジックを別の低頻度 cron（0 * * * * など）に分離
- fetchMs 改善: 外部APIの並列フェッチ化、タイムアウト上限の設定

## 完了条件

- `grandTotalMs` が 90秒以下に収まること
- 「実行時間超過」WARN の発生が停止すること
- PATH_B2 AbortError の発生頻度が有意に減少すること
