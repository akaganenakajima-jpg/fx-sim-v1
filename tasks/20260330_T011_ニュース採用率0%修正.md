# T011 ニュース採用率 0% 修正

**作成日**: 2026-03-30
**優先度**: 🔴 高
**発見経緯**: E2E Cycle 8〜10（バックエンドE2Eチェック）

---

## 背景・目的

`haiku_accepted=1` のレコードが 2026-03-29 16:01 UTC 以降ゼロ。
毎分「採用率 0%: 0件採用 / 30件処理」が system_logs に記録され続けている。
ニュースドリブンのエントリートリガーが完全に機能停止している状態。

## 現象

```
直近2時間: haiku_accepted=1 → 0件
haiku_accepted=-1 → 約101件/時（未採点 or スキップ）
haiku_accepted=0  → 約32件/時（フィルタ非通過）
```

system_logs（毎分連続）:
```
採用率 0%: 0件採用 / 30件処理
```

## 推定原因

1. `gemini-2.5-flash` がフィルタモデルとして旧モデル（`gemini-2.0-flash`）より選別が厳格
2. ニュース記事が全て「既読・重複・低スコア・FX無関係」と判定されている
3. `src/news.ts` の composite_score 採用閾値が高すぎる

## 調査手順

1. `src/news.ts` のニュースフィルタロジックを確認
   - `composite_score` の採用閾値（何点以上で haiku_accepted=1 になるか）
   - 重複排除ロジック（既読判定の条件）
   - gemini-2.5-flash へのフィルタプロンプト内容
2. 直近の news_raw で composite_score 分布を確認
   ```sql
   SELECT composite_score, COUNT(*) FROM news_raw
   WHERE fetched_at >= datetime('now','-6 hours')
   GROUP BY composite_score ORDER BY composite_score DESC LIMIT 20
   ```
3. reject_reason の分布を確認
   ```sql
   SELECT reject_reason, COUNT(*) FROM news_raw
   WHERE fetched_at >= datetime('now','-6 hours') AND haiku_accepted != 1
   GROUP BY reject_reason ORDER BY COUNT(*) DESC
   ```

## 修正方針（調査結果により決定）

- 閾値が高すぎる場合 → composite_score の採用下限を緩和
- 重複排除が過剰な場合 → 重複判定の TTL または類似度閾値を調整
- プロンプトが厳格すぎる場合 → gemini-2.5-flash 向けにプロンプト調整

## 完了条件

- `haiku_accepted=1` が直近1時間で1件以上記録されること
- system_logs の採用率 WARN が「0%: 0件」以外になること
