| カラム | 型 | NULL | デフォルト | 説明 |
|--------|------|------|-----------|------|
| id | INTEGER | NO | AUTOINCREMENT | 主キー |
| pair | TEXT | NO | 'USD/JPY' | 銘柄名 |
| direction | TEXT | NO | — | 'BUY' \| 'SELL' |
| entry_rate | REAL | NO | — | エントリーレート |
| tp_rate | REAL | YES | NULL | 利確レート（トレイリングで更新あり） |
| sl_rate | REAL | YES | NULL | 損切レート（トレイリングで更新あり） |
| lot | REAL | NO | 1.0 | ロットサイズ |
| status | TEXT | NO | 'OPEN' | 'OPEN' \| 'CLOSED' |
| pnl | REAL | YES | NULL | 損益（円換算、クローズ時に確定） |
| entry_at | TEXT | NO | — | エントリー日時（ISO 8601） |
| closed_at | TEXT | YES | NULL | クローズ日時（ISO 8601） |
| close_rate | REAL | YES | NULL | クローズレート |
| close_reason | TEXT | YES | NULL | 'TP' \| 'SL' \| 'TRAILING' \| 'MANUAL' |
| source | TEXT | YES | 'paper' | 'paper' \| 'oanda' |
| oanda_trade_id | TEXT | YES | NULL | OANDA トレード ID（実弾時のみ） |
| realized_rr | REAL | YES | NULL | 実現リスクリワード比（クローズ時に確定） |
| log_return | REAL | YES | NULL | 対数リターン（シャープレシオ計算用） |
| strategy | TEXT | YES | NULL | 使用戦略名（'mean_reversion' 等） |
| regime | TEXT | YES | NULL | 市場レジーム（'TREND' \| 'RANGE' 等） |
| session | TEXT | YES | NULL | 市場セッション（'TOKYO' \| 'LONDON' 等） |
| confidence | REAL | YES | NULL | エントリー確信度（0〜1） |
| partial_closed_lot | REAL | YES | NULL | 分割決済済みロット（TP1 到達分） |
| original_lot | REAL | YES | NULL | 元ロット（分割決済前） |
| tp1_hit | INTEGER | YES | NULL | TP1 到達フラグ（0/1） |
| trigger | TEXT | YES | NULL | エントリートリガー（'NEWS' \| 'LOGIC' \| 'RATE'） |
| mae | REAL | YES | NULL | 最大逆行幅（Maximum Adverse Excursion） |
| mfe | REAL | YES | NULL | 最大順行幅（Maximum Favorable Excursion） |
| original_sl_rate | REAL | YES | NULL | 初期 SL レート（トレイリング前の原点） |

**インデックス**:
```sql
CREATE INDEX idx_positions_status ON positions(status);
CREATE INDEX idx_positions_pair_status ON positions(pair, status);
```

**PnL 計算式**:
```
BUY:  pnl = (close_rate - entry_rate) × pnlMultiplier
SELL: pnl = (entry_rate - close_rate) × pnlMultiplier

実現RR計算式:
BUY:  realized_rr = (close_rate - entry_rate) / (entry_rate - sl_rate)
SELL: realized_rr = (entry_rate - close_rate) / (sl_rate - entry_rate)

勝敗判定: realized_rr >= 1.0 → 勝ち（WIN）, < 1.0 → 負け（LOSE）
```

**ライフサイクル**:
```
OPEN → [TP到達] → CLOSED (close_reason='TP')
     → [SL到達] → CLOSED (close_reason='SL')
     → [トレイリングSL] → CLOSED (close_reason='TRAILING')
```

**パージ**: なし（全件永続保持）

```sql

DELETE FROM system_logs
  WHERE id NOT IN (SELECT id FROM system_logs ORDER BY id DESC LIMIT 5000);

-- ② 時刻ベース（30日以上前を削除）
DELETE FROM system_logs WHERE created_at < datetime('now', '-30 days');
```
両方の条件を `runDailyTasks()` (daily-workflow.ts) で毎日 15:00 UTC に実行。

**TP/SL 計算式（参考）**:
```
最終TP幅 = ATR × atr_tp_multiplier × vix_tp_scale（VIX > vix_max×0.7 時のみ適用）
最終SL幅 = ATR × atr_sl_multiplier × vix_sl_scale × macro_sl_scale（条件分岐）
エントリー可否 = signal_strength(RSI偏差+ER) ≥ min_signal_strength
戦略選択 = strategy_primary → trend_follow 時は ER ≥ adx_min/100 を追加チェック
BB Breakout = isBBBreakout=true 時 → er_upper_limit スキップ、trend_follow スコアリング強制
```

| `decisions` | 〜40,000 件 | 90日超分の削除を検討 |

### 4.3 二重防衛パージの設計思想

`system_logs` と `news_fetch_log` に採用している「ID上限 + 時刻TTL」の組合せは以下の意図がある:

```
① ID上限（LIMIT 5000）:
   突発的な大量ログ発生時に即時キャップ。
   行数が爆発しても上限以上には増えない。

② 時刻TTL（30日）:
   IDが欠番だらけの場合でも古いデータを確実に削除。
   ストレージをリークさせない。

→ どちらか一方が失敗しても、もう一方が効いている二重安全網。
```

## 5. クエリパフォーマンス・メモリ保護

Cloudflare Workers は最大メモリ **128 MB**。D1 から大量行を返すと OOM になりうるため、API 関数は全て `LIMIT` 句でガードしている。

### 5.1 エンドポイント別 LIMIT 設定

| エンドポイント | クエリ対象テーブル | LIMIT | 実装関数 |
|--------------|-----------------|-------|---------|

| `GET /api/logs` | `system_logs` | **30件** | `getApiLogs()` |
| `GET /api/news` (acceptedNews) | `news_raw` (filter_accepted=1) | **30件** | `getApiNews()` |
| `GET /api/news` (newsTriggers) | `news_trigger_log` (直近12h) | **10件** | `getApiNews()` |
| `GET /api/status` (recentDecisions) | `decisions` | **20件** | `getApiStatus()` |
| `GET /api/status` (param_review) | `param_review_log` | **30件** | `getApiStatus()` |
| `GET /api/status` (recentCloses) | `positions` (CLOSED) | **50件** | `getApiStatus()` |
| `GET /api/rotation/history` | `rotation_log` | **20件** | index.ts handler |
| `GET /api/scores` | `stock_scores` | **60件** | index.ts handler |

### 5.2 SL パターン分析のバッファ制限

日次バッチの SL パターン分析（`runDailyTasks()`）では `LIMIT 500` で分析対象を制限:
```sql
SELECT p.close_reason, p.closed_at, p.pair, d.vix
FROM positions p
LEFT JOIN decisions d ON d.pair = p.pair
  AND d.created_at <= p.closed_at
WHERE p.status = 'CLOSED' AND p.close_reason IS NOT NULL
ORDER BY p.closed_at DESC
LIMIT 500
```
