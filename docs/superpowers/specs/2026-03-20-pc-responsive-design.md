# FX Sim PC版 レスポンシブデザイン設計書

> UX心理学 × Apple HIG 準拠 | FHD / 2K / 4K 対応 | Edge / Chrome 前提

## 1. 概要

モバイルファースト（375px）で構築された現UIを、FHD（1920×1080）/ 2K（2560×1440）/ 4K（3840×2160）のデスクトップ環境に最適化する。ブラウザchrome（タブ＋アドレスバー＋お気に入りバー ≈ 130px）表示状態を前提とする。

### 目標

- モバイルUIとの一貫性を維持しつつ、PC画面の広さを活用
- UX心理学43法則から12項目を適用し、理論に基づく設計判断
- Apple HIG（SF Pro Typography, 8ptグリッド, System Colors）準拠
- FHD〜4Kの解像度差（4倍）を clamp() ベースで滑らかに吸収

### 現状のUI構成

- モバイル4タブ: 資産 / AI判断 / 統計 / ログ（底タブバー）
- テーマ切替: ヘッダー右上のボタン（☀/🌙）
- ティッカーバー: 画面最上部のスクロール価格表示
- コンパクトサマリー: ティッカー下の資産残高要約行
- TP/SLバナー: 画面上部にフルワイドで表示
- ニュースドロワー: 画面下部から引き上げるドロワー

## 2. レイアウト構造

### 構成: 左サイドバー + メインコンテンツ + 右サイドパネル

```
┌─────────────────────────────────────────────────┐
│ Browser Chrome (tabs + address + bookmarks)     │
├──┬──────────────────────────────────┬────────────┤
│  │                                  │            │
│左│         メインコンテンツ          │  右サイド   │
│サ│   ┌─────────────────────────┐    │  パネル    │
│イ│   │ PnLヒーローカード       │    │            │
│ド│   └─────────────────────────┘    │  (コンテ   │
│バ│   ┌─────────────────────────┐    │   キスト   │
│ー│   │ ウォッチリスト（テーブル）│    │   連動)    │
│  │   └─────────────────────────┘    │            │
│  │                                  │            │
├──┴──────────────────────────────────┴────────────┤
```

**左サイドバー**: Fitts's Lawに基づく「無限エッジ」ターゲット。マウスを画面左端に投げるだけでタブにヒット。ブラウザchromeの直下に水平タブバーを重ねると認知負荷の「外在的負荷」が増大するため、縦配置を採用。

**左サイドバー（アイコンのみ）**: 幅56〜72pxのため、テキストラベルは8px以下の補助サイズで表示。主にアイコンで認識。
- タブ配置（上→下）: 📊資産 → 🤖AI → 📈統計 → 📋ログ → [spacer] → ⚙設定
- 系列位置効果: 最重要「資産」を最上部（初頭効果）、「設定」を最下部（新近効果）
- 「設定」はモバイルのヘッダー右上テーマ切替ボタンをサイドバー最下部に移動したもの。新規タブではなくテーマ切替・設定ポップオーバーを開く

**モバイル既存要素のPC版対応**:
- **ティッカーバー**: PC版では非表示（ウォッチリストのテーブルで全銘柄の現在値が常時見えるため不要）
- **コンパクトサマリー**: PC版では非表示（PnLヒーローカードが常時表示されるため不要）
- **TP/SLバナー**: メインコンテンツ領域の上部に表示（サイドバー・パネルは跨がない）。トースト通知としても右上に表示
- **ニュースドロワー**: PC版では右サイドパネルのニュースフィードに統合（ドロワーUI不使用）

| ブレークポイント | レイアウト | 説明 |

| 769〜1279px | タブレット | 上部水平タブバー（44px高）＋メインコンテンツ1カラム＋サイドパネルなし。ティッカー・コンパクトサマリー非表示。ウォッチリストはテーブル表示。max-width: 960px、中央寄せ |

- FHD: 1920×950px
- 2K: 2560×1310px
- 4K: 3840×2030px（OSスケーリング適用後は1920〜2560px相当が一般的）

## 4. 三段階スケーリング戦略（FHD / 2K / 4K）

Windows環境では4KモニタにOSスケーリング（150〜200%）が適用されるのが一般的。ネイティブ100%表示や2K 100%のユーザーにも対応するため、CSSビューポート幅に応じた clamp() ベースの滑らかなスケーリングを採用。

### スケーリング値一覧

| 要素 | FHD (1920px) | 2K (2560px) | 4K (3840px) | CSS実装 |
|---|---|---|---|---|
| コンテナ max-width | 1600px | 2000px | 2200px | `clamp(1200px, 85vw, 2200px)` |
| 左サイドバー幅 | 56px | 64px | 72px | `clamp(56px, 3.5vw, 72px)` |
| 右サイドパネル幅 | 320px | 380px | 420px | `clamp(300px, 18vw, 420px)` |
| PnLヒーロー | 42px | 52px | 64px | `clamp(2.625rem, 3vw, 4rem)` |
| セクションタイトル | 15px | 17px | 18px | `clamp(0.938rem, 1vw, 1.125rem)` |
| 本文・数値 | 14px | 15px | 16px | `clamp(0.875rem, 0.8vw + 4px, 1rem)` |
| キャプション | 11px | 12px | 13px | `clamp(0.688rem, 0.6vw + 4px, 0.813rem)` |
| ウォッチリスト行高さ | 40px | 44px | 52px | `clamp(40px, 2.5vw, 52px)` |
| カード padding | 16px 20px | 20px 24px | 24px 32px | `clamp(1rem, 1.2vw, 2rem)` |
| スパークライン幅 | 60px | 80px | 100px | `clamp(60px, 4vw, 100px)` |

### 4K CSS幅の注記

Windows 10/11 では4Kモニタに150〜200%スケーリングがデフォルト適用されるため、ブラウザの `window.innerWidth` は1920〜2560px相当になることが多い。ネイティブ100%（CSS幅=3840px）で使用するユーザーには clamp() の上限値が適用される。

### 2560px以上の追加レイアウト

- **2カラムウォッチリスト**: 保有ポジションと待機銘柄を横2列に配置。垂直スクロールを削減し全銘柄を一覧表示
- **拡張サイドパネル**: サイドパネル内にミニチャート（資産推移）を常時表示。段階的開示の閾値を広い画面で引き上げ

## 5. 右サイドパネル — コンテキスト連動

右パネルの内容はアクティブタブに応じて自動切替。ユーザーの現在の目標に関連する情報だけを提示し、無関連情報による注意の分散を防ぐ（選択的注意§4）。

| アクティブタブ | 右パネルの内容 | 理論根拠 |
|---|---|---|
| 資産 | ニュースフィード＋マーケット概況 | 選択的注意§4: 保有銘柄に影響するニュースに集中 |
| AI判断 | 判定履歴リスト＋スコアランキング | 系列位置§6: 直近判断を時系列で追跡 |
| 統計 | 銘柄詳細（クリック連動）＋取引履歴 | 段階的開示§2: 概要→詳細のドリルダウン |
| ログ | RiskGuard状態＋システム情報 | 認知負荷§1: 運用監視に必要な情報を集約 |

## 6. UX心理学 設計判断一覧（12項目）

### 6.1 認知負荷§1 — 情報密度の適応
- **適用箇所**: レイアウト全体 / ブレークポイント
- モバイル4タブ→PC2〜3カラム。画面サイズに応じてチャンク数を調整。1画面のウィジェットは7個以下を維持

### 6.2 視覚的階層§3 — F-pattern
- **適用箇所**: メインコンテンツのレイアウト
- 左上にPnLヒーロー（視覚的アンカー§3.1）。F-patternに沿って資産→ウォッチリスト→詳細へ視線を誘導

### 6.3 段階的開示§2 — タブ＋サイドパネル
- **適用箇所**: タブ構造 / ウォッチリスト行展開
- メイン情報は即座に表示、詳細はタブ切替またはクリック展開で開示。80%ルールに基づく初期表示

### 6.4 選択的注意§4 — コンテキスト連動パネル
- **適用箇所**: 右サイドパネル
- 現タブと無関連な情報を非表示にし、注意資源を目的に集中。バナーブラインドネス§5の回避にも貢献

### 6.5 系列位置効果§6 — サイドバータブ配置
- **適用箇所**: 左サイドバー
- 最重要「資産」を最上部（初頭効果）、「設定」を最下部（新近効果）。中間にAI判断・統計・ログを配置

### 6.6 損失回避§4（choice.md） — PnL表示
- **適用箇所**: ヒーローカード / ウォッチリスト
- 損失は利得の2倍強く感じる。損失時は赤色を抑制しフォントウェイトを軽く。利益時は緑を強調し達成感を演出

### 6.7 ドハティ閾値§8 — タブ遷移
- **適用箇所**: タブ切替アニメーション
- タブ切替は200ms以内のフェード遷移。400ms以下でフロー状態を維持。スケルトンは不要（データはキャッシュ済み）

### 6.8 美的ユーザビリティ§1（impression.md）
- **適用箇所**: 全体ビジュアル
- 8ptグリッド厳守。SF Pro Typography。Apple HIG System Colors。一貫したborder-radius（14px/10px/6px）で洗練された印象

### 6.9 ピークエンドルール§4（impression.md）
- **適用箇所**: PnLヒーロー / TP達成バナー
- 資産残高のヒーロー表示がページの「ピーク」。TP達成時の祝福バナーが「エンド」。体験の印象を両端で設計

### 6.10 変動型報酬§2（motivation.md）
- **適用箇所**: リアルタイム価格更新
- ウォッチリストの価格・PnLが予測不能に変動。スパークラインの視覚フィードバックが習慣的チェックを促進

### 6.11 目標勾配§1（motivation.md）
- **適用箇所**: ROI表示 / 勝率表示
- ROIと勝率がゴール指標として機能。数値が上がるほどモニタリング頻度が加速する効果を活用

### 6.12 デフォルトバイアス§7 — 初期タブ
- **適用箇所**: アプリ起動時
- 「資産」タブをデフォルト表示。最も頻繁に確認する情報を暗黙の推奨として設定。変更の手間を排除

## 8. PC固有インタラクション

### 8.1 ウォッチリスト行ホバー
マウスホバーで行ハイライト＋右端に「詳細→」ボタン出現。クリックでボトムシートではなくサイドパネルに詳細表示。

### 8.2 ニュース行クリック
クリックでインライン展開（影響分析テキスト表示）。モバイルのドロワーではなくサイドパネル内で完結。

### 8.3 キーボードショートカット
1-4キーでタブ切替。Rで更新。Tでテーマ切替。パワーユーザーの操作速度を加速。
- **入力フィールドフォーカス時は無効化**: `document.activeElement` が input/textarea/select の場合はショートカットを無視
- ブラウザ既定のショートカット（Ctrl+R等）とは競合しない（修飾キーなしの単体キーのみ使用）

### 8.4 ウォッチリスト テーブル表示
モバイルのカード列表示から、PCではテーブル形式に変更。列ヘッダーで情報の対応関係が明確になり認知負荷を低減。

## 9. カラーシステム

既存CSS変数を再利用し、PC専用の新規追加は最小限にする。

**既存CSS変数（そのまま使用）**:
- `--bg`, `--bg-elevated`, `--bg-secondary`, `--label`, `--label-secondary`, `--separator`
- `--accent` (#0A84FF), `--green` (#30D158), `--red` (#FF453A), `--orange` (#FF9F0A)
- `--radius` (12px), `--radius-sm` (8px)

**PC専用に追加するCSS変数**:
| 変数名 | ダークモード | ライトモード | 用途 |
|---|---|---|---|
| `--sidebar-bg` | var(--bg) | var(--bg) | サイドバー背景 |
| `--panel-bg` | var(--bg-elevated) | var(--bg-elevated) | 右パネル背景 |
| `--panel-border` | var(--separator) | var(--separator) | パネル区切り線 |

**border-radius**: 既存の `--radius` (12px) / `--radius-sm` (8px) を統一的に使用。設計モックアップの14px/10px/6pxは既存変数に合わせて12px/8px/4pxに修正。

**ライトモード**: 既存の `[data-theme="light"]` CSS変数がそのまま適用される。PC専用変数も `var()` で既存変数を参照するため、テーマ切替に自動追従。

## 10. HTML構造の変更

PC版では以下のHTML要素を `dashboard.ts` に追加する:

1. **左サイドバー** (`<nav class="pc-sidebar">`): メインコンテナの先頭に追加。モバイルでは `display:none`
2. **右サイドパネル** (`<aside class="pc-panel">`): メインコンテナの末尾に追加。モバイル・タブレットでは `display:none`
3. **ウォッチリストのテーブルマークアップ**: JS で `window.innerWidth >= 769` 時にカード→テーブルに動的切替

既存のモバイルUI要素（底タブバー、ティッカー、ドロワー等）はHTMLに残し、メディアクエリで表示/非表示を制御。

## 11. 実装上の制約

- 既存のモバイルUI（style.css.ts, app.js.ts, dashboard.ts）にCSSメディアクエリを追加する形で実装
- メディアクエリ構成: `@media (min-width: 769px)`, `@media (min-width: 1280px)`, `@media (min-width: 1920px)`, `@media (min-width: 2560px)`
- 新規ファイル作成は不要（既存ファイルへの追記のみ）
- JavaScript の変更: サイドバータブ切替、ウォッチリストのテーブル表示切替、キーボードショートカット、右パネルのコンテキスト連動
- 既存の API レスポンス構造は変更しない
- `viewport` メタタグ: PC表示時に `user-scalable=no` を削除し、ブラウザズームを許可（JSで動的に変更）

## 12. エッジケース

- **ブラウザズーム**: ズームにより `window.innerWidth` が変化しブレークポイントが遷移する（FHD 150%ズーム → 1280px相当 → デスクトップレイアウト）。これは意図した動作として許容
- **ウルトラワイド（21:9/32:9）**: `max-width: clamp(...)` によりコンテンツは中央に収まり、左右に余白が生まれる。意図した動作として許容
- **最小ウィンドウサイズ**: 768px以下に縮小するとモバイルレイアウトに自動フォールバック。横スクロールは発生しない
- **OSスケーリング変更**: clamp() ベースのため、スケーリング変更後のリロードで即座に適切なサイズに調整
