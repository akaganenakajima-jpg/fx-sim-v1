### 2.2 静的ファイル配信と Workers の役割分担

| リクエスト | 処理主体 | 説明 |
|-----------|---------|------|
| `GET /app.js` | `[assets]` (CDN) | Workers を経由しない。Cloudflare エッジから直接配信 |
| `GET /style.css` | `[assets]` (CDN) | 同上 |
| `GET /` | Workers | `getDashboardHtml()` が HTML 殻を返す。`<script src="/app.js">` を含む |
| `GET /api/*` | Workers | D1 クエリ結果を JSON で返す |

`wrangler.toml` の `[assets]` バインディング:
```toml
[assets]
directory = "./public"
```

`public/` ディレクトリのファイルはビルド成果物ではなく**手書きのバニラJS/CSS**。バンドラー不使用。

**Ph.6 以降のAI呼出パターン**:
- **毎分のAI判定（Path A）は廃止**。テクニカル判定は Logic が自律的に実行
- **AI の役割**: ① ニュース理解（Path B, newsStage1/newsStage2）、② N トレードごとのパラメーター最適化（Param Review）

## 3. モジュール仕様

### 3.1 index.ts — エントリーポイント

**cron フロー（Ph.6 以降）**:
1. ワンタイムマイグレーション確認
2. `Promise.allSettled` で news / indicators / frankfurter を並列取得
3. ニュースソース統計を `news_fetch_log` にバッチ INSERT
4. 全銘柄の現在レートをマッピング（Yahoo 失敗時はキャッシュフォールバック）
5. 全ポジションの TP/SL チェック + トレイリングストップ
6. **Logic エントリー**: RSI + ER + ATR + 拡張パラメーター（vix_tp_scale 等）で自律判定
7. スキップ銘柄をバッチ INSERT（HOLD 記録）
8. BUY/SELL ならブローカー経由でポジション開設
9. **Path B**（ニュースハッシュ変化時のみ）: AI ニュース分析 → EMERGENCY/TREND 処理
10. **Param Review**（N 件取引ごと）: AI がパラメーターを最適化提案
11. 実行時間記録、30s 超過で WARN、ログパージ、日次サマリー

**HTTP ハンドラ**: 17 ルート（静的殻1 + PWA4 + API12）

### 3.2 gemini.ts — AI API

**プロンプト設計**:
- `newsStage1`: `thinkingBudget: 0` でシンキングトークン無効化（高速化）、ブラックリスト方式（「迷ったら採用」）
- `newsStage2`: 採用済みニュースを精査して EMERGENCY / TREND_INFLUENCE に分類
- Param Review: 取引実績データ + 現行パラメーターを渡し最適化提案を取得

### 3.2b logic-trading.ts — Logic エントリー（Ph.3〜6）

テクニカル指標のみでエントリー判定を行う（AI 不使用）。

- `evaluateEntry()`: RSI + ER + ATR で BUY/SELL/HOLD を判定
- TP/SL: `ATR × atr_tp_multiplier × vix_tp_scale`（VIX 条件分岐）
- SL: `× vix_sl_scale × macro_sl_scale`（マクロ下落時）
- `min_signal_strength` フィルタ: RSI偏差 + ER の平均が閾値未満ならスキップ
- `strategy_primary`: `trend_follow` 時は ER ≥ adx_min/100 を追加チェック
- **`isBBBreakout` フラグ**: `true` の場合 `er_upper_limit` チェックをスキップし、`overrideTrendFollow=true` で trend_follow スコアリングを強制適用（BB Breakout 戦略対応）

### 3.2c param-review.ts — AI パラメーターレビュー（Ph.4〜6）

N 件取引到達時に AI が全パラメーター（基本 10 + 拡張 5）を最適化提案。

- `runParamReview()`: 銘柄ごとにレビュートリガー判定 → AI 呼出 → 検証 → 反映
- `validateAndClamp()`: AI 提案値を ±20% 範囲 + 絶対上下限でクランプ
- `strategy_primary`: ホワイトリスト検証（'mean_reversion' | 'trend_follow'）

### 3.3 position.ts — ポジション管理

- `checkAndCloseAllPositions()`: 全 OPEN ポジションの TP/SL を一括チェック
- `openPosition()`: 新規ポジション作成（D1 INSERT）
- **トレイリングストップ**: `trailingActivation` 以上の含み益で SL を `trailingDistance` 幅に引上げ

### 3.4 broker.ts — ブローカー抽象化

**BrokerAdapter インターフェース**:
```typescript
interface BrokerAdapter {
  openPosition(params: OrderParams): Promise<BrokerResult>;
  closePosition(params: CloseParams): Promise<BrokerResult>;
  updateStopLoss(params: UpdateSLParams): Promise<BrokerResult>;
}
```

- **PaperBroker**: 何もしない（D1 記録は呼出元が行う）
- **OandaBroker**: OANDA REST API v20 で MARKET 注文（FOK）
- **getBroker()**: `TRADING_ENABLED` + 銘柄設定で自動ルーティング
- **withFallback()**: OANDA 失敗時にペーパーにフォールバック

### 3.5 risk-guard.ts — RiskGuard

4 段階チェック（全て通過で `allowed: true`）:

1. **日次損失キルスイッチ**: 当日の実弾損失合計が上限超過 → 全停止
2. **最大ポジション数**: 実弾 OPEN が上限到達 → 新規拒否
3. **異常レート検知**: 前回比乖離率が閾値超過 → 発注拒否
4. **ロット制限**: 超過時はクランプ（`adjustedLot`）

### 3.6 instruments.ts — 銘柄定義

**46 銘柄アクティブ**（FX4 + 円クロス3 + 株価指数6 + 暗号1 + コモディティ5 + 日本株19 + 米国株8）

`InstrumentConfig` の主要フィールド:
- `broker`: 'paper' | 'oanda' | 'alpaca'
- `tier`: 'A' | 'B' | 'C' | 'D'（ティア別ロット倍率で資金配分）
- `correlationGroup`: 相関グループ（同グループ同方向 2 件以上でブロック）
- `rateChangeTh`: フィルタ閾値
- `pnlMultiplier`: PnL 円換算倍率
- `tpSlMin` / `tpSlMax` / `rrMax`: サニティ上下限
- `trailingActivation` / `trailingDistance`: トレイリングストップ設定
- `assetClass`: 'forex' | 'cfd' | 'stock'（株式の場合）
- `stockSymbol` / `tradingHoursJST` / `tradingHoursET`: 株式用設定

### 3.7 filter.ts — フィルタ

**Ph.6 以降**: `shouldCallGemini()` は廃止。Logic エントリーが毎分実行されるため、AI 呼出頻度の制御は不要。

`isSkipSchedule()`: 重要指標発表時間帯の判定（news-trigger 側で参照予定）

- 早朝禁止帯: 18:00〜22:00 UTC（JST 3:00〜7:00）

### 4.2 データフロー — 遅延ロードアーキテクチャ

初期描画を爆速化するため、エンドポイントをタブ別に分割し「必要な時だけ取得」する。

```
初期ロード時:
  app.js 実行 → refresh() → GET /api/status → render(data) → HOMEタブ描画

タブ切替時（初回のみ）:
  switchTab('tab-stats')
    ├── activeTabId = 'tab-stats'  ← 同期的に即更新（Race Condition防止）
    └── !historyData → GET /api/history → historyData にキャッシュ
          └── if (activeTabId === 'tab-stats') → Statsタブ再描画

  switchTab('tab-news' | 'tab-ai')
    ├── activeTabId = タブID
    └── !newsData → GET /api/news → newsData にキャッシュ
          └── if (activeTabId === ...) → 再描画

  switchTab('tab-log')
    ├── activeTabId = 'tab-log'
    └── !logsData → GET /api/logs → logsData にキャッシュ
          └── if (activeTabId === 'tab-log') → 再描画

  switchTab('tab-strategy')
    ├── activeTabId = 'tab-strategy'
    └── !paramsData → GET /api/params → paramsData にキャッシュ
          └── if (activeTabId === 'tab-strategy') → 再描画

30秒ポーリング（バックグラウンドリフレッシュ）:
  refresh()
    ├── GET /api/status → render(data)   ← 常に実行
    ├── GET /api/news → render(data)     ← 常にバックグラウンドリフレッシュ
    └── アクティブタブに応じて追加リフレッシュ:
          tab-stats     → GET /api/history
          tab-log       → GET /api/logs
          tab-strategy  → GET /api/params
```

**ポーリング間隔一覧**:

| 対象 | 間隔 | 定数 |
|------|------|------|
| `/api/status` | 30秒 | `setInterval(refresh, 30000)` |
| `/api/params` | 60秒 | `PARAM_POLL_MS = 60000` |
| `/api/rotation` + `/api/scores` | 60秒 | `ROT_POLL_MS = 60000` |
| `/api/history`, `/api/logs`, `/api/news` | タブ切替時 + 30秒ポーリングに乗り合わせ | — |

**モジュールレベル状態変数**:

```javascript
var lastData      = null;  // /api/status の最新レスポンス
var activeTabId   = 'tab-home';  // Race Condition 防止用（後述）
var historyData   = null;  // /api/history キャッシュ
var logsData      = null;  // /api/logs キャッシュ
var newsData      = null;  // /api/news キャッシュ
var paramsData    = null;  // /api/params キャッシュ（tab-strategy 遅延ロード）
```

### 4.3 非同期通信の堅牢化

#### Race Condition 防止 — activeTabId ガード

タブを素早く切り替えたとき、古いタブの非同期レスポンスが後から返って現在のタブを上書きする問題を防ぐ。

**仕組み**:
1. `switchTab(tabId)` の **先頭で同期的に** `activeTabId = tabId` を更新
2. 非同期コールバック内で `if (activeTabId === tabId)` を確認してから描画

```javascript
function switchTab(tabId) {
  activeTabId = tabId;  // ← ① まず同期的に更新
  // ...
  if (tabId === 'tab-stats' && !historyData) {
    loadHistory().then(function() {
      if (activeTabId === 'tab-stats' && lastData)  // ← ② 確認してから描画
        render(lastData);
    });
  }
}
```

**例**: A → B → A と切り替えた場合、B用の`loadHistory()`が遅れて返ってきても、その時点で `activeTabId === 'tab-home'` なので描画はスキップされる。

#### Stale State 防止 — バックグラウンドリフレッシュ

30秒ポーリングは `/api/status` のみでは、非HOMEタブ表示中にデータが古くなる問題があった。

**解決策**: ポーリングのたびに、アクティブタブに対応するエンドポイントも背後でリフレッシュ。

```javascript
function refresh() {
  fetch('/api/status').then(function(data) {
    render(data);
    // ニュースは常にバックグラウンドリフレッシュ
    loadNews().then(function() { if (lastData) render(lastData); }).catch(function() {});
    // アクティブタブのデータも更新
    if (activeTabId === 'tab-stats') {
      loadHistory().then(function() {
        if (activeTabId === 'tab-stats' && lastData) render(lastData);
      }).catch(function() {});
    } else if (activeTabId === 'tab-log') {
      // ... 同様
    }
  });
}
setInterval(refresh, 30000);
```

### 4.4 主要関数（app.js）

| 関数 | 役割 |
|------|------|

## 6. 注意事項

- **cron 予算制御**: Workers 有料プラン（30s CPU 上限）。ニュース分析フォールバックも `getAdaptiveB2Timeout()` で適応制御（8〜20s）
- **API キーローテーション**: AI 呼出は Path B + Param Review のみ。5 Gemini + 2 OpenAI + 1 Anthropic キーを巡回
- **D1 バッチ最適化**: スキップ銘柄の HOLD 記録、レート更新は `env.DB.batch()` で一括
- **PnL 単位**: 全て円換算（`pnlMultiplier` で変換）。S&P500: ×10、US10Y: ×5000 等
- **Path A 廃止（Ph.6）**: 毎分のAI銘柄判定を廃止。テクニカル判定は Logic が拡張パラメーターで自律実行
- **週末制約（⚠️ 変更禁止）**: FX市場クローズ中（金曜21:00〜日曜22:00 UTC）は暗号資産（BTC/ETH/SOL）のみ取引可。取引判断関数は `getTradeableInstruments(INSTRUMENTS, weekendStatus)` を経由すること（`src/weekend.ts`）。詳細は `CLAUDE.md §週末市場クローズ制約` を参照
- **テスト実行**: `npm test` で vitest が起動。`src/weekend.test.ts` が週末制約の回帰テスト（12件）を担う
- **`trigger` フィールドの信頼性**: 戦略分類には `strategy` フィールドより `trigger` フィールドを優先すること。BB Breakout エントリーでも `strategy='mean_reversion'` が格納される設計のため（classifyStrategyType 参照）
